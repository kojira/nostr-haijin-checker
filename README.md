# nostr-haijin-checker 💀

`npub` を 1 つ渡すと、その人の **「廃人度（Haijin score）」** を 0〜100 点で
採点します。CLI でも Web（ブラウザ）でも使えます。次の **3 つの軸**で採点します。

- **短期アクティブ度** … 直近どれだけ活発か
- **利用パターン** … 常時稼働度・連投傾向・交流密度（リプライ/リアクション/リポスト）
- **長期継続・古参度** … どれだけ長く続けているか

そのうえで **総合廃人スコア・ランク・各スコアの根拠**、および
**観測信頼度 (confidence)** を出力します。

> 「廃人（はいじん）」＝ ここでは *Nostr に時間を溶かしているヘビーユーザー* を
> 面白がるためのネットスラング。娯楽・自己診断向けのツールであり、
> 医学的・心理学的な依存度判定ではありません。

---

## 特徴

- **説明可能**: すべてのスコアに「なぜその点数なのか」の根拠（reason）が付きます。
- **ローカル完結**: リレーへ直接 WebSocket 接続。API キーやサーバー不要。
- **過去へ遡る取得**: 時間ウィンドウを適応的に分割して掘り直すため、投稿が多い
  ユーザーでも取りこぼしにくいです（[取得方針](#取得方針適応的タイムウィンドウ) を参照）。
- **複数リレーを並行取得**: 1 つのリレーが失敗してもほかのリレーから取得を続けます。
  取得状況は `history` / `notes` に記録します（[タイムアウト設計](#タイムアウト設計グローバルではなくリクエストリレー単位) を参照）。
- **取得進捗の可視化**: 取得中の応答リレー数・件数・遡れた最古などをライブ表示します。
- **連続実稼働日数（ストリーク）**: 連続して投稿のあった日数を数えます。**メインの取得で
  集めたイベントから導出**し（別経路の追加取得はしません）、連続日数は「連続実稼働」シグナル
  （長期軸・重み 12%）として総合スコアに加点され、連続日数が長いほどスコアが上がります
  （[詳細](#連続実稼働日数ストリーク取得済みイベントから導出)）。
- **CLI / Web で採点ロジックを共有**: 取得（`src/nostr/`）と採点（`src/scoring/`）が独立。
- **投稿用テキスト生成**: 診断結果から、総合スコア・ランク・連続実稼働・主な根拠をまとめた
  短い日本語テキストを生成します。Web ではワンクリックでクリップボードへコピーでき、
  Nostr / SNS にそのまま貼れます（[詳細](#投稿用テキストの生成と共有)）。
- **JSON 出力**: `--json` でプログラム連携可能。

> 取得対象の kind 一覧や署名検証の扱い（relay-trusted）などの仕様は、下部の
> [取得するイベント種別](#取得するイベント種別-kind) / [制約・限界](#制約限界-limitations) を参照してください。

## 使い方

### 基本

```bash
# ビルド後（推奨）
node dist/index.js <npub>

npm run dev -- <npub>
```

`<npub>` は `npub1...` 形式、または 64 桁の hex 公開鍵を受け付けます。

### オプション

| オプション | 説明 | デフォルト |
| --- | --- | --- |
| `-r, --relays <urls>` | カンマ区切りのリレー URL。未指定ならデフォルト一式 | （内蔵リレー） |
| `--initial-window <sec>` | 最初に範囲を区切る粗いウィンドウ幅（秒） | `2592000`（30 日） |
| `--dense-threshold <n>` | この件数以上を返したウィンドウは中点で分割して掘り直す（密判定） | `1000` |
| `--min-window <sec>` | これ以下の幅のウィンドウはそれ以上分割しない（秒） | `3600`（1 時間） |
| `--max-windows <n>` | 1 リレーあたりのウィンドウ処理数の安全上限 | `5000` |
| `--max-events <n>` | 取得イベント総数の上限（`0` で無制限） | `0` |
| `--since <unixsec>` | この時刻 (UNIX 秒) より古いイベントは取りに行かない（下限） | `1609459200`（2021-01-01） |
| `-t, --tz <hours>` | 時間分布の判定に使う UTC オフセット（時間） | `9`（JST） |
| `--window-timeout <ms>` | **1 リクエスト**（1 リレー × 1 ウィンドウ）のタイムアウト。超えたウィンドウだけ中断し他リレーに影響しない | `30000`（30 秒） |
| `--relay-timeout <ms>` | **1 リレー**の総取得時間の上限。超えたリレーだけ打ち切り、他リレーは継続する | `120000`（120 秒） |
| `--overall-timeout <ms>` | 任意の**全体安全上限**（`0` で無効）。主たる打ち切りではなく暴走防止の二次的な保険 | `0`（無効） |
| `--timeout <ms>` | **[非推奨]** 旧グローバルタイムアウト。`--overall-timeout`（安全上限）の別名として解釈される | （未指定） |
| `--no-streak` | 連続実稼働日数（ストリーク）を計測しない（取得済みイベントからの導出を行わない）（[詳細](#連続実稼働日数ストリーク取得済みイベントから導出)） | （計測する） |
| `--json` | JSON で出力 | off |
| `-h, --help` | ヘルプ表示 | |
| `-V, --version` | バージョン表示 | |

> 旧版の `-l, --limit`（リレーごとの件数上限）は廃止されました。取得は
> nostr-fetch による**適応的タイムウィンドウ**に変わり、件数ベースの
> `--page-size` / `--max-pages` も廃止しました。深さは「どこまで古い時刻まで
> （`--since`）」「密なウィンドウの分割しきい値（`--dense-threshold`）」
> 「ウィンドウ数の安全上限（`--max-windows`）」「総件数の上限（`--max-events`）」で
> 制御します。
>
> 旧版の `--late-start` / `--late-end`（深夜帯の指定）も廃止しました。「深夜が多い＝
> 廃人」という粗い基準をやめ、**稼働時間帯の広さ（常時稼働度）** を測る設計に
> 変えたため、深夜帯のしきい値指定は不要になりました（[採点ロジック](#採点ロジックスコアリング) を参照）。

#### タイムアウト設計（グローバルではなくリクエスト／リレー単位）

取得は各リレーを**独立に**処理します。これに合わせ、タイムアウトも単一のグローバル
期限ではなく、**責務ごとに 3 段へ分割**しています。**あるリレー（またはウィンドウ）が
時間切れになっても、他のリレーは一切止まりません**（NostrActivity の「タイムアウトは
リクエスト／リレー単位に閉じる」設計に倣っています）。

| 段 | オプション | 何を縛るか | 既定 |
| --- | --- | --- | --- |
| リクエスト／ウィンドウ | `--window-timeout` | 1 リレー × 1 ウィンドウの `fetchAllEvents` 1 回 | `30000`（30 秒） |
| リレー | `--relay-timeout` | 1 リレーが全ウィンドウ処理に費やせる総時間 | `120000`（120 秒） |
| 全体（任意） | `--overall-timeout` | 暴走防止の安全上限（`0` で無効） | `0`（無効） |

- **ウィンドウタイムアウトはグローバル停止ではありません。** 各ウィンドウは独立に計時され、
  期限を超えた**そのウィンドウだけ**を中断します（実装は `fetchAllEvents` 専用の
  `AbortController` ＋ `Promise.race` で wall-clock を必ず縛り、ハングしません）。
- **リレータイムアウトはそのリレーだけを打ち切ります。** 超過したリレーは `timeout` 状態で
  終わり、それまでに集めた分は捨てません（部分成功）。他リレーは取得を続行します。
- **全体安全上限は二次的な保険**です。主たる打ち切りはあくまでウィンドウ／リレー単位で、
  `--overall-timeout` は既定では無効（`0`）です。発火しても暴走防止の最終手段にすぎません。
- 旧版は 12〜15 秒の**単一グローバル期限で全リレーを一括中断**していたため、500 件級の
  ページ取得には小さすぎました。新既定はリクエスト／リレー単位で現実的な値に引き上げています。
- 失敗・タイムアウトしたリレーは `history.relayStats`（`status: "timeout"`）と `notes` に
  正直に残ります。

#### 取得中の進捗表示

取得中は stderr に進捗行を**ライブ更新**します（`--json` 時は出しません）。

```
取得中... リレー 5/7 応答 ・ 1638 件 ・ 8 ウィンドウ ・ 最古 2022-12-16 ・ 2.6s
```

- 応答／失敗したリレー数、これまでに集めたユニーク件数、処理した総ウィンドウ数、
  ここまで遡れた最古の日付、経過時間を数値で表示します。
- 各リレーは独立に取得するため、**1 つが落ちても進捗は止まりません**。
  失敗したリレー数は進捗行・最終出力（`取得:` 行）・`notes` に反映されます。

#### 取得方針（適応的タイムウィンドウ）

取得基盤は [nostr-fetch](https://github.com/jiftechnify/nostr-fetch) です。
フィルタは **authors ＋ 許可した kind（`ALLOWED_KINDS`）** でリレーへ問い合わせ、
`fetchAllEvents(relays, filter, {since, until})` を「1 つの時間ウィンドウ」の
プリミティブとして使います。`[since, until]`（`--since` 〜 現在）を
`--initial-window`（既定 30 日）でタイル分割し、各ウィンドウを取得します。
署名検証は行わず、リレーが返したイベントをそのまま信頼します（relay-trusted）。

**密なウィンドウは中点で再帰分割**します。あるウィンドウが `--dense-threshold`
（既定 1000）件以上を返し、かつ幅が `--min-window`（既定 1 時間）より広ければ、
中点で 2 つの子ウィンドウに分割して掘り直します。これは、リレーが 1 リクエストで
全件を返し切れない（リレー側のキャップ）場合でも取りこぼさないための保険です。
`fetchAllEvents` 自体もウィンドウ内で内部ページング・重複排除を行うため、
1 ウィンドウの取得は境界安全で、件数境界で起きうる取りこぼしを避けられます。

**リレーは 1 つずつ独立にウィンドウ処理**し、結果をグローバルに id で重複排除します
（同一 id が複数リレー／重なるウィンドウから来ても 1 件）。あるリレーが接続失敗・
タイムアウトしても**そのリレーだけを打ち切り、残りのリレーは取得を続行**します
（部分継続）。**リレーの役割分割（長期用/短期用）は行いません**。リレー個別の到達状況
（応答／失敗／遡れた最古／処理ウィンドウ数）は `history.relayStats` に記録されます。

各リレーは次のいずれかに達するまで処理します。

- **要求範囲 `[since, until]` を全ウィンドウ覆い切る**（正常終了 / `ok`）
- `--max-windows` に達する（`maxWindows`）
- `--max-events` に達する（`maxEvents`）
- `--window-timeout` / `--relay-timeout`（または任意の `--overall-timeout`）を超える（`timeout`。
  ただし**そのリレーだけ**が打ち切られ、他リレーは継続する）

許可リスト（`ALLOWED_KINDS`）の全イベントを取得するのは、kind1 以外（リアクション kind7・
リポスト kind6・公開チャンネル・DM 関連など）だけの日も**実稼働日・継続性に算入する**ためです。
フォローリスト（kind3）など許可リスト外の kind は取得・算入しません。

#### 連続実稼働日数（ストリーク）— 取得済みイベントから導出

上記の**取得（適応的タイムウィンドウ取得）**で集めたイベントは総合スコアの採点に使います
（密度・連投・交流・時間帯分布など、1 件単位の特徴を全部使うため、許可リスト
（`ALLOWED_KINDS`）の全イベントを掘ります）。

**連続実稼働日数（ストリーク）**の判定に必要なのは **「その日に投稿が 1 件でもあるか」** だけ
です。これは上の取得で集めたイベントから導けるため、**別経路の追加取得は一切行いません**。
旧実装はストリーク専用に「日ごとに最新 1 件だけを遡る軽量プローブ（`fetchLastEvent`）」を
リレーへ投げていましたが、二重取得になり両者の観測範囲がずれると結果が食い違う余地があった
ため、**取得済みイベントからの導出**（`src/scoring/streak.ts` の `deriveStreak`・純関数）に
一本化しました。二重取得と結果ズレの両方を取り除いています。

- **実稼働日 = ローカル日（`--tz` 補正後の暦日）に 1 件以上イベントがある日**。
  1 日に何件投稿していても、その日は **1 実稼働日**として数えます。
- 取得済みイベント（許可 kind のみ）をローカル日単位に集計し、**最新の実稼働日**から
  1 日ずつ過去へ。実稼働日が連続する限りカウントし、活動の無い日（ギャップ）に当たった
  瞬間に終端します。内部に焼き込んだ日数の天井はありません（3 年超の長いストリークも
  そのまま数え切ります）。
- **`truncated` は取得の完全性で決まります**。取得が要求範囲 `[since, until]` を覆い切れた
  （`HistoryMeta.historyComplete===true`）なら、観測したギャップは確定なので `truncated=false`
  ＝連続日数は確定値です。取得が掘り切れていない（各種上限・タイムアウト＝`false`）場合だけ、
  観測できた連続日数は実際の **下限**にとどまる可能性があるため `truncated=true` として
  控えめに扱います。
- 得られた連続日数は「連続実稼働」シグナル（長期軸・重み 12%）として
  **総合スコア（`totalScore`）に加点**され、連続日数が長いほどスコアが上がります
  （[採点ロジック](#採点ロジックスコアリング) の表を参照）。

ストリークの加点設計（説明可能性のため明示します）:

- **スコア化**: `score = saturating(currentStreakDays, 60)`（早く伸びて約 60 日で頭打ち。
  例: 7日 ≈ 51 / 14日 ≈ 66 / 30日 ≈ 84 / 60日以上 = 100）。
- **重み**: 固定 `0.12`。短期の活発さ（重み 0.30）を支配しない modest な大きさにしています。
  longTerm と違い**観測信頼度では割り引きません**（取得済みイベントから連続日を直接導出済みのため）。
- **truncated の扱い**: 取得が掘り切れていない（`historyComplete===false`）ときの
  `currentStreakDays` は「実際の連続日数の **下限**」です。`saturating` は単調増加なので、
  この下限から得た加点は過大主張ではなく **控えめな下限**になります。既知の長いストリークは
  高く出つつ、正確な天井は断定しません（reason には「≥」を併記）。
- **連続日数 0**: 加点 0（合算には固定重みで参加するため、ストリークが弱いほど総合は下振れします）。
- ストリークを計測しない（`--no-streak` / Web のオフ）場合は、このシグナルは
  `signals` にも合算・正規化にも一切現れません。

結果（`StreakInfo`）は次を含みます。

| フィールド | 意味 |
| --- | --- |
| `currentStreakDays` | 連続実稼働日数（最新の実稼働日から途切れずに遡れた日数） |
| `lastActiveDay` | 最新の実稼働日（ローカル日）の `YYYY-MM-DD`。活動が無ければ `null` |
| `daysSinceLastActive` | 最新実稼働日が「今日」から何日前か（`0`=今日, `1`=昨日…） |
| `ongoing` | 継続中か（最新実稼働日が今日 or 昨日なら `true`） |
| `observedActiveDays` | ストリーク算出の母数となった観測実稼働日数（取得イベントから導いた distinct な実稼働日数） |
| `truncated` | 取得が掘り切れていない（`historyComplete===false`）ために下限にとどまるか。`true` なら実際はさらに長い可能性 |

ストリークは**取得済みイベントから導出**するため、別経路のネットワーク取得・専用タイムアウト・
専用上限を持ちません。`--no-streak`（Web のオフ）で計測を無効化できます。CLI・Web 版とも
同じ `deriveStreak`（環境非依存の純関数）を共有します。

## 採点ロジック（スコアリング）

総合スコアは各シグナルの **信頼度を考慮した重み付き平均**（0〜100）です。
基本は 5 シグナル、ストリーク（連続実稼働）を取得したときは 6 シグナルになります。
各シグナルは 3 つの軸（短期 / パターン / 長期）に属します。

| シグナル | 軸 | ベース重み | 何を見るか | 100点の目安 |
| --- | --- | --- | --- | --- |
| 短期アクティブ度 (`shortTermActivity`) | 短期 | 30% | 観測ウィンドウ内の密度（件/日）＋稼働日率。サンプル少なら割引 | 高密度・ほぼ毎日 |
| 常時稼働度 (`temporalCoverage`) | パターン | 20% | 1 日の中で投稿が広い時間帯に分散しているか（時間帯カバレッジ＋昼夜分散＋分布の均一さ） | 24 時間まんべんなく投稿 |
| 連投傾向 (`bursts`) | パターン | 15% | 連投（5 件以上を 120 秒以内）に含まれる投稿割合 | 全体の 40% 以上 |
| 交流密度 (`engagement`) | パターン | 15% | リプライ＋リアクション＋リポストの割合 | 全体の 60% 以上 |
| 長期継続・古参度 (`longTermRetention`) | 長期 | 20%※ | 初観測からの経過 × 観測ウィンドウ長 × 実稼働日数。**観測信頼度で割引** | 長期間・多数の実稼働日 |
| 連続実稼働 (`streakRetention`) | 長期 | 12%† | 連続実稼働日数（取得済みイベントから導出）。`saturating(日数, 60)`。**信頼度割引なし** | 連続約 60 日以上 |

> ※ 長期軸（古参度）の重みは **ベース 20% × 観測信頼度** です。観測ウィンドウが短いと
> 信頼度が 0 に近づき、総合スコアからの寄与も自動的に小さくなります。
>
> † 連続実稼働シグナルは**ストリークを計測したときだけ**現れる任意シグナルです（`--no-streak`
> や Web のオフでは合算・正規化・`signals` のいずれにも現れません）。連続日数が長いほど
> スコアが上がり、約 60 日で頭打ちになります。取得が掘り切れていない（truncated）時は下限として
> 控えめに加点します。詳細は[ストリークの加点設計](#連続実稼働日数ストリーク取得済みイベントから導出)。

### 各シグナルの正規化

- **短期アクティブ度** … `密度スコア(0.6) + 稼働日率スコア(0.4)` を、サンプル信頼度
  `log10(1+件数)/log10(1+30)` で割り引き。密度スコアは飽和カーブ
  `100 · log10(1 + 件/日) / log10(1 + 25)`。短い観測でも密度が高ければ高く出ますが、
  これは「短期」の軸に閉じ込められます。
- **常時稼働度** … 投稿の時刻分布（ローカル 0–23 時）から、3 つの観点を合成します。
  `score = 100 · (0.5·カバレッジ + 0.3·昼夜区分カバレッジ + 0.2·均一さ)`。
  - **カバレッジ** = 投稿のあった distinct な時間帯数 / 24（稼働時間帯の広さ）。
  - **昼夜区分カバレッジ** = 6 時間ごと 4 区分（深夜/午前/午後/夜）のうち活動した区分数 / 4。
  - **均一さ** = 時刻ヒストグラムの正規化シャノンエントロピー（1 = 24 時間に完全均一）。
  → 「深夜に集中」より「**一日中まんべんなく＝いつ見ても居る**」ほど高くなります。
  夜型/朝型といった単なる生活リズムの違いを廃人度と取り違えません。
- **連投・交流** … 「目安割合に対する比率 × 100」を 0〜100 にクランプ。
- **長期継続・古参度** … 素の古参度
  `age(0.35) + span(0.30) + activeDays(0.35)`（各 saturating）を、
  **観測信頼度 = clamp01(ウィンドウ長/180) × clamp01(実稼働日数/30)** で割引。
  ウィンドウ < 45 日 または 実稼働 < 8 日のときは `longTermAssessable=false` とし、長期継続よりも短期の活発さを中心に評価します。
- **連続実稼働（ストリーク）** … 取得済みイベントから導出した連続実稼働日数を
  `saturating(日数, 60) = 100 · log10(1 + 日数) / log10(1 + 60)` でスコア化（早く伸びて約 60 日で頭打ち）。
  観測信頼度では割り引かず（取得済みイベントから連続日を直接導出済みのため）、固定重み 12% で合算します。
  truncated（取得が掘り切れていない）時の日数は **実際の連続日数の下限**で、`saturating` が単調なため加点も控えめな下限になります。

### 観測メタ情報（`observation`）

| フィールド | 意味 |
| --- | --- |
| `observedWindowDays` | 観測ウィンドウの長さ（最古〜最新投稿の日数） |
| `firstSeenAgeDays` | 最古の観測投稿が「現在」から何日前か（古参度の材料） |
| `observedActiveDays` | 投稿があった実日数（distinct） |
| `confidence` | 観測信頼度 0–1（ウィンドウ長 × 実稼働日数） |
| `longTermAssessable` | 長期継続を評価できるだけの観測があるか（真偽） |

### 総合スコアの合算（confidence-aware）

短期＋パターンは満額、長期（古参度）は「ベース重み × 信頼度」で重み付けし、
**実際に観測できた分の重みだけで正規化**します。ストリークを取得したときは、
固定重み（信頼度割引なし）の項を**分子・分母の両方**に足します。

```
streakW = ストリークあり ? 0.12 : 0
total = ( Σ(短期+パターンの score×weight)
          + 古参度score × 0.20
          + 連続実稼働score × streakW )
        / ( 0.80 + 0.20 × confidence + streakW )
```

これにより、観測が短いユーザーを「長期不明」で不当に減点せず、かつ古参を僭称することも
ありません。0〜100 の正規化も保たれ（分母にも同じ重みを足すため）、ストリークがあるときは
**連続日数が長い（`連続実稼働score` が高い）ほど分子だけが増えて総合スコアが上がります**。
ストリークが無いときは `streakW = 0` となり、式は従来と完全に一致します。

> 飽和・クランプ・観測信頼度の実装は `src/scoring/signals.ts`、重みと合算は
> `src/scoring/index.ts`（`WEIGHTS`）にあります。閾値はすべてコード上の定数で、
> 調整しやすいようにしています。

### ランク

総合スコアを以下の段階にマッピングします（`src/scoring/rank.ts`）。

| スコア | ランク |
| --- | --- |
| 85–100 | 💀 完全体廃人 |
| 70–84 | 🔥 廃人 |
| 55–69 | ⚡ ヘビーユーザー |
| 40–54 | 🌱 アクティブ |
| 20–39 | 🍵 ライトユーザー |
| 0–19 | 😴 サイレント・休眠 |

### 取得するイベント種別 (kind)

リレーへの問い合わせは、**許可リスト（`ALLOWED_KINDS`）の kind だけ**を取得します
（フィルタは authors ＋ kinds）。許可リストは `src/kinds.ts` で定義し、取得
（`src/nostr/query.ts`）・ストリーク導出（`src/scoring/streak.ts`）・採点（`src/scoring/`）が
共通で参照します。

| kind | 種別 | カテゴリ |
| --- | --- | --- |
| `0` | プロフィール（メタデータ） | プロフィール |
| `1` | テキスト投稿（`e` タグ付きはリプライ扱い） | 投稿 |
| `4` | 暗号化 DM（NIP-04） | DM 関連 |
| `6` | リポスト | リポスト |
| `7` | リアクション | リアクション |
| `40` / `41` / `42` / `43` / `44` | 公開チャンネル作成・メタデータ・発言・非表示・ミュート（NIP-28） | 公開チャンネル |
| `13` / `14` / `1059` | シール・チャットメッセージ・ギフトラップ（NIP-17/59） | DM 関連 |

稼働日・継続性の判定はこの許可リストの kind を対象にするため、kind1 以外
（リアクション kind7・リポスト kind6・公開チャンネル・DM 関連など）だけの日も実稼働日に数えます。

採点シグナルの算出では、主に次の kind を意味づけして使います。

- `kind 1` … テキスト投稿（`e` タグ付きはリプライ扱い）
- `kind 6` … リポスト
- `kind 7` … リアクション

> **許可リスト外の kind（フォローリスト kind3、リレーリストなど）は取得もせず、
> 密度・稼働日・継続性のいずれにも算入しません。** 採点入力（`scoreEvents`）でも
> 許可外 kind は防御的に除外されます。
>
> 取得イベントの**署名検証は行いません**。リレーが返したイベントをそのまま信頼します
> （relay-trusted。[制約・限界](#制約限界-limitations) を参照）。

```
  短期アクティブ度　　 ████████████████░░░░  80
  長期継続・古参度　　 ███████████████████░  97
  利用パターン　　　　 █████████████████░░░  83
```

→ 観測ウィンドウが長く `confidence=100%`。長期継続・古参度が高く出ます。

### B) 短期だけ集中（新規・短期中心で評価）

直近 7 日だけ猛烈に投稿しているユーザーの例。**短期は満点で、長期継続よりも短期の活発さを中心に評価します**:

```
=== Nostr 廃人度チェック ===
npub: npub1example...newbie
観測: 210 件 / 2023-11-14 15:13Z 〜 2023-11-20 18:40Z (JST)
観測ウィンドウ: 6.1 日 / 実稼働 7 日 / 初観測から 7.4 日前  観測信頼度 1%

スコア（3 軸）:
  短期アクティブ度　　 ████████████████████ 100
  長期継続・古参度　　 ░░░░░░░░░░░░░░░░░░░░   0  短期中心で評価
  利用パターン　　　　 ████████░░░░░░░░░░░░  40

総合廃人スコア: 62 / 100
ランク: ⚡ ヘビーユーザー
        毎日しっかり投稿。立派な常連。
        ※ 長期継続よりも短期の活発さを中心に評価しています。

注意:
  - 観測ウィンドウが 6.1 日（実稼働 7 日）のため、短期の活発さを中心に評価しています。
  - 取得はリレー側の保持期間・件数制限に依存します。実際の活動の一部しか観測できていない可能性があります。
```

→ **A と B は短期では同等でも、長期継続では明確に区別されます。**
旧版のように「7 日の観測を 3 年の継続と取り違える」ことはありません。

> 注: 取得は複数リレーへ適応的タイムウィンドウで問い合わせ、id で重複排除します。
> どこまで遡れたか（履歴を掘り切れたか）は出力末尾の「取得:」行と `notes`、
> JSON では `history` フィールドで確認できます。深く遡るには `--since` を緩める／
> `--max-windows` / `--max-events` / `--relay-timeout` / `--window-timeout` を増やしてください。

`history` は取得（適応的タイムウィンドウ）のメタ情報です。取得経路を介さず

> 互換のため一部フィールド名は旧称のままです（意味が変わったものは下表に明記）。

| `oldestCreatedAt` / `newestCreatedAt` | 観測できた最古／最新の投稿時刻（UNIX 秒。0 件なら null） |
| `relaysQueried` | 問い合わせたリレー数 |
| `relaysSucceeded` / `relaysFailed` | 応答した／失敗・タイムアウトしたリレー数（部分継続の指標） |
| `relayStats` | リレー個別の取得結果（`url` / `status` / `events` / `pages`(=ウィンドウ数) / `oldestReached` / `error`） |
| `elapsedMs` | 取得にかかった時間 |
| `hitEventCap` | 件数上限（`maxEvents`）に当たったか |
| `hitPageCap` | ウィンドウ数上限（`maxWindows`）に当たったか（旧称のまま） |
| `timedOut` | タイムアウト／中断したか |
| `noProgress` | 廃止（常に `false`） |

> `relayStats[].status` は `ok`（全ウィンドウ処理完了／件数上限での正常打ち切り）/
> `empty`（このリレーには投稿なし）/ `failed`（接続失敗）/ `timeout`（時間切れ）/
> `maxWindows`（ウィンドウ数上限）のいずれかです。
> `failed` / `timeout` のリレーがあっても、残りのリレーから取得は継続されます。

> **「complete」と「incomplete」の意味**: `historyComplete=true` は、健全なリレーが
> 要求範囲 `[--since, 現在]` を全ウィンドウ覆い切った（`stopReason==="ok"`）ことを表す
> **best-effort の見込み**です。リレーは履歴の完全性を保証しないため、`ok` でも
> 「観測できた最古」が**本当の初投稿とは限りません**（リレーが古いイベントを破棄
> している可能性、または `--since` の床より前にも投稿がある可能性）。一方
> `historyComplete=false`（ウィンドウ数上限・件数上限・タイムアウト・全リレー失敗で
> 打ち切り）のときは、それより前にも投稿がある可能性が高く、長期継続・古参度は
> **過小評価され得ます**。いずれの場合も `notes` に正直な但し書きが入ります。

## Web 版（ブラウザ / GitHub Pages）

ブラウザで `npub` を入力すると、リレーへ直接接続して総合スコア・ランク・各シグナルの
根拠を表示する静的フロントエンドを同梱しています。GitHub Pages にそのままデプロイできます。

### Web の入力オプションと進捗表示

「詳細設定」（折りたたみ）から CLI 相当のパラメータを調整できます。

| UI 項目 | 対応する CLI | 既定値 |
| --- | --- | --- |
| Relays | `--relays` | 内蔵リレー |
| 分割しきい値（密判定の件数） | `--dense-threshold` | `1000` |
| 最大ウィンドウ数（遡る深さ） | `--max-windows` | `5000` |
| Timezone | `--tz` | `9` |
| ウィンドウ取得タイムアウト | `--window-timeout` | `30000`(ms) |
| リレー総取得タイムアウト | `--relay-timeout` | `120000`(ms) |
| 全体安全上限（0 で無効） | `--overall-timeout` | `0`（無効） |

> タイムアウトは**グローバル（全体一括停止）ではなくリクエスト／リレー単位**です。
> あるリレー（またはウィンドウ）が時間切れになっても、**他のリレーは取得を続行**します
> （[タイムアウト設計](#タイムアウト設計グローバルではなくリクエストリレー単位) を参照）。

取得中は、フォームの下に**ライブ進捗パネル**を表示します（`onProgress` で逐次更新）。

- 大きな数値で **取得イベント数 / 応答リレー数 / 失敗リレー数 / 取得ウィンドウ数** を表示。
- **ここまで遡れた最古の日時・経過時間・完了リレー数**を併記。
- **リレー個別の状態リスト**（取得中／完了／投稿なし／接続失敗／時間切れ／ウィンドウ上限…）を
  色分けして逐次更新。1 つのリレーが失敗しても他はそのまま進みます。
- 完了時はステータス欄に
  `完了: <件数> 件を採点（応答 a/b リレー・<ウィンドウ数> ウィンドウ・履歴 …）。` を表示します。
  履歴部分は掘り切れたかで文言が変わります
  — 掘り切れた場合は **「リレーが返す限界まで到達」**、
  途中で打ち切った場合は **「掘り切れず（<停止理由>）」**。

### 投稿用テキストの生成と共有

採点が終わると、結果の末尾に **「投稿用テキスト」カード**を表示します。
そのまま Nostr / SNS に貼り付けられる、短く自然な日本語の共有文です。

- 含める情報: **総合スコア + ランク**（絵文字 + ラベル）、**連続実稼働日数**（ストリーク）、
  **主な根拠**（影響度＝`score × weight` の高いシグナルを重複排除して上位抜粋）、ハッシュタグ。
- ストリークが無い（未計測 / 0 日）場合は、その行を省いた有用なテキストになります。
- **「投稿テキストをコピー」**ボタンで [Clipboard API](https://developer.mozilla.org/docs/Web/API/Clipboard/writeText)
  へコピーします。API が使えない環境では `textarea` の選択コピーへ穏当にフォールバックし、
  成否を短く表示します。

整形ロジックは `src/scoring/shareText.ts` の純関数 `buildShareText(result, opts?)` に
切り出してあり、CLI 側（`src/scoring/index.ts` から再エクスポート）でも再利用できます。
ハッシュタグや根拠の表示件数は `opts`（`hashtag` / `maxSignals`）で調整できます。

```ts
import { buildShareText } from "./src/scoring/index.js";

const text = buildShareText(result); // result: ScoreResult
// 例:
//   Nostr廃人度 72/100 🔥 廃人
//   🔥 連続実稼働 30日（継続中）
//   主な根拠: 常時稼働度 92 ・ 短期アクティブ度 78 ・ 交流密度 64
//   #Nostr廃人度チェッカー
```

### NIP-07（ブラウザ拡張）で自分を採点

npub を手で入力する代わりに、**NIP-07 対応のブラウザ拡張**
（[nos2x](https://github.com/fiatjaf/nos2x) / [Alby](https://getalby.com/) など）から
公開鍵を取り込んで、その場で自分を採点できます。

1. NIP-07 対応の拡張機能をブラウザに導入し、鍵を設定しておく。
2. ページの **「NIP-07 で自分の公開鍵を使う」** ボタンを押す。
3. 拡張機能が `window.nostr.getPublicKey()` を通じて公開鍵を返すと、
   手入力なしでそのまま採点フローが走ります（取得した npub は入力欄にも反映されます）。

> 取得するのは**公開鍵のみ**で、署名や秘密鍵は一切要求しません。
> 拡張機能が見つからない場合は、ステータス欄にその旨を表示し、
> npub の直接入力（従来どおり）にフォールバックできます。

> 取得は CLI と同じ `src/nostr/` を再利用しますが、WebSocket 実装だけを
> 環境ごとに差し替えています（Node は `ws`、ブラウザはネイティブ `WebSocket`）。
> 共通の取得ロジックは `src/nostr/query.ts` に集約し、
> `fetch.ts`（Node）/ `fetch.browser.ts`（ブラウザ）が薄くラップしています。

1. GitHub でリポジトリの **Settings → Pages** を開く。
2. **Build and deployment → Source** を **「GitHub Actions」** に設定する。
3. `main` ブランチに push する（または Actions タブから手動実行）。
4. ワークフローが `npm run web:build` を実行し、`dist-web/` を Pages に公開します。
   公開 URL は `https://<ユーザー名>.github.io/<リポジトリ名>/` です。

> アセットは相対パス（`vite.config.ts` の `base: "./"`）で出力するため、
> リポジトリ名やサブパスに依存せずそのまま動作します。

#### 方法 B: 手動デプロイ

```bash
npm run web:build
# dist-web/ の中身を gh-pages ブランチや任意の静的ホスティングへ配置
```

### ブラウザ版の前提

ブラウザから直接リレーへ接続するため、いくつか細かな前提があります。

- HTTPS 配信のため、リレーは `wss://` のみ利用できます（UI 側で `wss://` 以外は弾きます）。
- 取得件数・保持期間はリレー依存で、古い投稿は取得できないことがあります。

## 制約・限界 (Limitations)

- **観測できるのは公開イベントの一部だけ**。リレーごとの保持期間・件数制限に
  依存し、古い投稿や別リレーの投稿は取得できないことがあります。
  → スコアは「観測できた範囲」に対する近似値です。
- **観測ウィンドウが短いときは短期の活発さを中心に評価します**。これは
  **設計上の安全装置**でもあります。短い観測から長期の継続を断定せず、
  `longTermAssessable=false` / `confidence` で明示します（短期の活発さは別軸で評価）。
- **短期アクティブ度は観測ウィンドウ「内」の密度**。過去の長い空白の有無に関わらず、
  「いま活発か」を表します（長期の継続とは分離）。
- **タイムゾーンは固定オフセット**（既定 JST=UTC+9）。ユーザーの実際の生活時間帯や
  サマータイムは考慮しません。`--tz` で調整してください。
- **署名検証は行いません（relay-trusted）**。リレーが返したイベントをそのまま信頼して
  採点します。偽装イベントを混ぜたリレーを使うとスコアは歪み得ます（信頼できるリレーの利用が前提）。
- **リアクション/リポストの取得網羅性はリレー依存**。交流密度は過小評価され得ます。
- **「リプライ」判定は `e` タグの有無のみ**（NIP-10 のマーカー種別までは見ない簡易版）。
- **サンプルが少ない（<30件）場合は `notes` で参考値である旨を補足します。**
- 娯楽・自己診断向けの MVP です。**精度より「動くこと・説明できること」を優先**しています。

## 今後の拡張余地（設計意図）

採点層（`src/scoring/`）は Nostr 取得に依存しない純粋なデータ→スコア変換として
切り出してあります。これにより:

- **Web フロント**: ✅ 実装済み（`src/web/`）。同じ採点層をブラウザでそのまま再利用し、
  GitHub Pages に静的デプロイできます。
- **LLM 連携**: `reason` 群を要約・コメント生成のプロンプト材料に利用可能。
- **シグナル追加**: `signals.ts` に関数を足し `WEIGHTS` に重みを追加するだけ。
  → CLI・Web の両方に自動で反映されます。
